INTRODUCTION
What Is This Comic About?
🌐 Universal Acceptance (UA) means that ALL valid domain names and email addresses — in ANY language, script, or length — work equally well in every app, website, and system on the Internet.
🗺️ Today, over 1,200 domain extensions exist (like .عرب, .中国, .london, .engineering). Many apps still reject them — that's a UA failure.
🌍 UA is the gateway to the next billion internet users who want to go online in their own language and script.
✅ This comic — created using ICANN's Universal Acceptance course — will show you exactly what UA is, why it matters, and how to fix it.

CHAPTER 1
The Definition of UA
🌐 Universal Acceptance ensures all valid domain names and email addresses are treated equally by all internet-enabled apps, devices, and systems — regardless of script, language, or character length.
📏 The old internet was built for short, Latin-script, 2–3 letter extensions like .com, .net, .org. But today there are 1,200+ extensions and IDNs in Arabic, Chinese, Hindi, and more.
🔤 Three pillars of UA: (1) Accept all valid domain names. (2) Accept all valid email addresses. (3) Process and display them correctly — in the right script!
💡 Key insight: UA is a TECHNICAL problem — not a language or culture problem. The internet CAN handle these addresses. Apps just need to be updated to use them properly.

CHAPTER 2
Internationalized Domain Names
🔤 IDNs are domain names using characters from non-Latin scripts — Arabic, Chinese, Cyrillic, Devanagari, Thai, and more. They give users the ability to navigate the internet in their own language.
🏷️ Two forms: The U-Label is what users type and see (e.g. 公司.中国). The A-Label is the ASCII version the DNS system actually uses (e.g. xn--55qx5d.xn--fiqs8s). Apps must handle both.
📛 LDH-Label: A traditional domain using only Letters, Digits, and Hyphens (a-z, 0-9, -). Most old apps are only designed to accept LDH-labels — that's the root of many UA failures.
⚠️ The problem: Over 60 IDN country-code TLDs exist (e.g. .السعودية, .ไทย, .中国). Apps that only validate for LDH will reject all of them, locking out entire communities.

CHAPTER 3
Email in Your Own Language
📧 EAI — Email Address Internationalization — is the set of standards that allows email addresses to use non-ASCII characters in BOTH the local part (before @) and the domain part (after @).
📬 An email address has two parts: local-part @ domain. EAI allows both to use any Unicode character — Arabic, Chinese, Hindi, accented Latin, and more.
🔧 Why apps fail: Most validation code uses simple rules like "only a-z, 0-9, dots, hyphens." This was written for ASCII-only email. EAI requires updated validation using Unicode-aware standards (RFC 6530–6532).
✅ UA-readiness includes EAI. A system is not UA-ready unless it correctly accepts, stores, processes, and displays internationalized email addresses.
🚧CHAPTER 4: THE THREE UA FAILURES
A UA problem shows up in THREE main ways. Let's look at each one.
❌ FAILURE 1: REJECTION
😤
"Invalid email" — when my email address IS valid by international standards!
App validation code uses regex patterns that only allow ASCII characters. Result: legitimate users locked out.
❌ FAILURE 2: WRONG DISPLAY
😕
I typed 公司.中国 in my browser but it shows xn--55qx5d.xn--fiqs8s. What happened?!
The app failed to convert A-label back to U-label for display. Users see confusing machine-readable codes.
❌ FAILURE 3: LENGTH REJECTION
🤦🏽
The app says my TLD ".engineering" is "too long to be real." But it IS real — it's a valid new gTLD!
Old code assumed TLDs are max 4 characters (.com, .info). New gTLDs can be much longer — apps must update their length rules.
A 2017 UASG study found UA-readiness is a $9.8 billion opportunity. Every UA failure is lost business — and a lost user.

CHAPTER 4
Three Types of UA Failure
❌ Rejection: The app refuses to accept a valid IDN or EAI address. Usually caused by outdated validation logic (regex) that only allows ASCII characters (a-z, 0-9, hyphens, dots).
❌ Wrong display: The app accepts the address internally but shows the A-label (punycode) instead of the U-label (readable script). This confuses users and erodes trust.
❌ Length rejection: Old code assumes TLDs are 2–4 characters (.uk, .com, .info). New gTLDs like .engineering or .solutions are longer and get incorrectly rejected.
💰 The cost of UA failures: UASG research identified UA-readiness as a $9.8+ billion opportunity. Every failure means lost customers, lost revenue, and users excluded from the digital economy.

🔧CHAPTER 6: HOW TO MAKE SYSTEMS UA-READY
Lin the developer is on a mission — make her company's app UA-ready!
👩🏻‍💻
Our sign-up form rejects half the world's internet users. Let's fix that — step by step.
🔧 STEP 1: FIX DOMAIN NAME VALIDATION
❌ OLD: if(domain matches /^[a-zA-Z0-9.-]+$/)
⬇️ update to
✅ NEW: Use a Unicode-aware DNS validation library
Allow A-labels (xn--...) and U-labels. Don't limit TLD length to 4 characters. Reject only truly invalid addresses.
🔧 STEP 2: FIX EMAIL VALIDATION
❌ OLD: only accepts [a-z0-9._%+-]+@[a-z0-9.-]+
⬇️ update to
✅ NEW: Accept Unicode in local-part AND domain
Follow RFC 6530–6532 standards. Use libraries like libeai or language-specific EAI validators.
🔧 STEP 3: TEST, STORE & DISPLAY CORRECTLY
✅ Store addresses in Unicode (UTF-8) — not ASCII-encoded ✅ Display U-labels to users (readable script, not punycode) ✅ Test with real IDN and EAI addresses — use UASG test tools ✅ Check: uasg.tech/eai-check for your email server
👩🏻‍💻
After these fixes, our app accepted Amina's Arabic address, Raj's Hindi email, AND my Chinese domain!
🎉 That's what being UA-ready looks like!
CHAPTER 6
The UA-Ready Checklist
✅ Domain validation: Accept all valid TLDs — including new gTLDs (any length) and IDN TLDs (non-ASCII). Use a standards-based library, not a hand-written regex with a hardcoded list of extensions.
✅ Email validation: Follow RFC 6530–6532. Allow Unicode characters in both the local-part and domain. Store in UTF-8. Never downgrade to ASCII without user consent.
✅ Display: Convert A-labels back to U-labels before showing users. A user who typed 公司.中国 should see 公司.中国 — not punycode gibberish.

A More Inclusive Internet
